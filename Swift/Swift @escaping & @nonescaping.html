Swift @escaping & @nonescaping

Swift @escaping & @nonescaping

Swift 中的闭包经常会被编译器提示加上 @escaping 关键字, 不知道你有没有想过为什么要加上这个呢?

结论: 如果闭包是异步执行,需要加上 @escaping 关键字, 如果是在方法执行完毕之前执行的闭包, 就无需加上 @escaping 关键字.

解释:
在方法中的闭包经常会出现这样一个场景, 比如一个网络请求 RequestApi 的回调闭包, 这样的闭包通常都是异步执行的, 等待接收到服务器的响应后, 再去执行闭包, 这时, 这个闭包是在 RequestApi 方法执行完毕后, 再去执行的, 这时, 我们可以看做这个闭包已经逃离了 RequestApi.比如下面的代码: 

func slCompare<T>(_ array:[T], handler: @escaping (T, T) -> Bool) {
  DispatchQueue.main.async {
    if handler(array[0], array[1]) {
      print("大于")
    } else {
      print("小于")
    }
  }

  print(array)
}


slCompare(a) { (a, b) -> Bool in
  return a < b
}

我们专门用了 GCD 的一个异步队列去执行这个闭包, 这个 handler 就必须加上 @escaping

如果我们去掉了 DispatchQueue.main.async {} 那么这个闭包就是跟随方法一起执行的, 而不是等待方法执行结束后, 再执行. 跟方法一起执行, 也就是跟方法同生共死的闭包, 就是非逃逸闭包, 不需要加上 @escaping.

生命周期:
@nonescaping closures

  1. 当函数调用的时候, 闭包当做函数参数传递给函数
  2. 和函数一起做一些额外的工作
  3. 函数运行闭包
  4. 函数告诉编译器返回.
